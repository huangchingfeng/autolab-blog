# 當產品經理開始寫程式：AI 如何讓職業邊界消失

> 本文是社群貼文的完整延伸版。
> 社群版講了核心概念，本文加上完整場景分析和企業應用觀點。

---

## 重點回顧

91APP 內部 20 幾個敏捷小團隊，因為 AI 的導入，出現了一個前所未有的現象：產品經理開始寫程式，工程師開始挑戰商業策略。職稱正在失去意義，職業邊界正在消失。

這不只是一家公司的故事。這是每一個企業接下來都會面對的組織變革。

### 當 PO 開始 Vibe Coding

91APP 的產品長 Happy 和技術主管 Spencer（前 iCHEF 共同創辦人）在塞掐 Side Chat 的訪談中，分享了一個讓我很有感觸的現象。

91APP 有 20 幾個敏捷小團隊，每個團隊 7 到 15 人。自從大家開始使用 AI 工具，產品經理不再只是寫規格書了。他們開始用 Vibe Coding 的方式，直接把想法做成可以跑的 prototype，然後拿去給客戶看。

工程師看了之後開始挑毛病：「這有資安問題」「效能不行」。可是 PO 反過來問了一個很好的問題：「這功能很獨立啊，測一下不行嗎？POC 跑一下看看，為什麼不行？」

更有趣的是反方向的變化。工程師開始 challenge PO 寫的規格書和商業策略。因為用 AI 快速做 prototype 之後，直接讓業務拿去跑，客戶的反饋馬上就驗證了這個需求到底有沒有價值。

Spencer 用了一句話總結：**「職業的邊界正在變模糊。」**

### Scrum 的預言終於成真

這裡有一個很微妙的點。Scrum 作為敏捷開發的方法論，其實一直有個準則：團隊裡的每個角色可以兼做別人的工作。但在現實中，這幾乎從來沒有真正發生過。因為跨界的學習成本太高了。

PO 怎麼可能會寫程式？RD 怎麼懂商業策略？

AI 改變了這個方程式。跨界的成本趨近於零。PO 不需要學三年程式設計，他只要會用 AI，就能做出可以跑的 prototype。RD 不需要讀 MBA，他用 AI 就能分析市場數據、產出商業報告。

以前那個「理想中的跨界」，現在突然變成了每天都在發生的事。

### 30 個團隊，30 種做法

Spencer 還提到一個很務實的觀察。91APP 有 30 個團隊（加上 iCHEF 合併後的團隊），每個團隊用 AI 的方式都不一樣。有的用 AI 加速開發，有的用 AI 做市場驗證，有的用 AI 改善客戶體驗。

他們沒有強制統一做法。

取而代之的是讓團隊互相分享，看看別人怎麼用、學到什麼。Spencer 說現在大家都處在一種「很興奮的狀態」——在找出新的可能性，但還沒有定性。

他把這比喻成 Anthropic、早期 Google、早期 Amazon 的黃金時期。那個時期的特色就是：沒有精細的分工、沒有嚴格的組織架構、有事情就搶著做，所有人朝著同一個方向衝。

### 做給 Agent 用，不是做給人用

Spencer 分享了一個很銳利的個人經驗。他做產品的第一版是給人類用的——漂亮的介面、好看的路徑、直覺的操作。

後來他覺得自己傻了。

「應該做給 Agent 用，不應該做給人用。」

Agent 不需要漂亮的 UI。它需要的是 API，甚至是更輕量的 CLI（命令列介面）。Agent 要的是效率，不是好看。

這個觀察背後有數據支持。過去半年，美國的 SaaS 軟體 ETF 整體下跌了 30% 到 50%。但如果把這些公司拆開來看，跌的是那些「按人頭收費」的軟體——像客服系統 Zendesk，因為客服人員正在被 AI 取代，席次制的收費模式面臨挑戰。

但做基礎建設的軟體——Cloudflare、發簡訊的服務、做身份認證的平台——這些「按 API 呼叫次數收費」的公司，營收反而往上升。

**誰在服務 Agent，誰就在成長。**

### MCP 可能會被拋棄？

訪談中還有一個很前沿的技術觀察。MCP（Model Context Protocol）是目前讓 AI 連接外部工具的主流協議，但 Spencer 認為它太重了——消耗大量的 Token，維護成本高，Server 常常掛掉。

他的預測是：CLI 可能會取代 MCP，成為 Agent 更偏好的介面。因為 CLI 更輕、更快、消耗更少的 Token。

這個觀點很大膽，但從技術角度來看，邏輯是通的。Agent 不需要人類友好的介面，它需要的是最高效的溝通方式。

## 【獨家】AI Agent 的進化：從工具到好員工

![Creative shot of a human brain model in a pot against a blue backdrop, symbolizing mind and thought.](images/ai-agent-tools.jpg)
*Photo by [Amel Uzunovic](https://www.pexels.com/@amel-uzunovic-440739273) on [Pexels](https://www.pexels.com/photo/top-view-of-a-brain-in-a-cup-15410081/)*


### 心跳機制：主動找事做的 Agent

Happy 在訪談中提到了一個叫 OpenClaw 的開源專案（有人叫它 OpenCloud），做了一件以前沒有人做的事：**給 AI Agent 裝上心跳（Heartbeat）**。

什麼是心跳？以前的 Agent，你不叫它，它就不動。N8N 之類的自動化工具，頂多是設定一個排程，每天早上幾點跑一次。

OpenClaw 的心跳機制不一樣。Agent 會定時去自己的記憶裡面翻找，主動尋找可以做的事情。你不用下指令，它就會想起來：「欸，主人昨天說要訂 2 月 14 的情人節餐廳，我來查一下價位。」

Spencer 的評價很精準：「以前的 Agent 是工具。現在的 Agent 是好員工。」

### 靈魂設定：每個 Agent 都不一樣

OpenClaw 還有一個「靈魂」的概念。開發者要設定一個叫 `Soul.md` 的檔案，定義 Agent 的人格、偏好、短期記憶、長期記憶。

這個設計很聰明。它把 prompt 做了分類——不是一大堆 system prompt 混在一起，而是有結構地定義一個 Agent「是什麼樣的人」。

結合心跳（定時主動出擊）和靈魂（個人化人格），AI Agent 正在從「被動回答問題的工具」進化成「主動執行任務的夥伴」。

## 阿峰觀點

![A modern humanoid robot with digital face and luminescent screen, symbolizing innovation in technology.](images/artificial-intelligence-business.jpg)
*Photo by [Tara Winstead](https://www.pexels.com/@tara-winstead) on [Pexels](https://www.pexels.com/photo/robot-pointing-on-a-wall-8386440/)*


我在企業培訓裡一直說：「你是機長，AI 是機組人員。」

91APP 的案例讓我看到，機組人員不只是升級了——整個機艙的分工方式都在重新定義。以前副駕駛就是副駕駛，空服員就是空服員。現在副駕駛開始做空服員的判斷，空服員開始挑戰副駕駛的航線建議。

這對每個工作者的啟示很清楚：

**不要被你的 title 限制住。** AI 讓跨界的成本趨近於零。你不需要等 IT 部門幫你做工具、不需要等數據團隊幫你跑報表、不需要等行銷部門幫你做素材。你可以自己用 AI 把事情做完。

問題不是 AI 會不會取代你。問題是你願不願意跨出自己的職稱，去做以前做不到的事。

Spencer 說他看到 91APP 內部「所有人都在搶事情做」。這才是正確的姿態。AI 不是來搶你的工作的，AI 是來讓你搶到更多工作的。
